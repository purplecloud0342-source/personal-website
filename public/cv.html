知至 — AI 知识管理工具多 Agent 浏览器扩展，从 0 到 1 2026.06 · GitHub

模型选型：为什么是多 Agent 而不是单一模型？最早用单一 LLM 做「一次性分析+拆解」，结果发现模型同时处理「理解内容」「归纳观点」「生成计划」时输出很泛、抓不住重点。拆成四个专职 Agent（Observer 追踪 → Analyst 拆解 → Kickstart 生成步骤 → Reflection 模式识别），每个只做一件事，Agent 间上下游引用。评测对比：多 Agent 输出的行动计划可执行率明显高于单模型——单模型容易跳过具体步骤直接给结论

提示词调优：Analyst Agent 早期 Prompt 只要求「总结核心观点」，产出和通用摘要没区别。迭代后明确要求「提取可行动的观点 + 标注观点来源的时间戳/段落」，观点可追溯性从模糊提升到每条观点都能定位原文。Kickstart Agent 的 Prompt 经历三轮迭代——从「生成学习计划」到「生成可执行的下一步 + 预估耗时 + 难度标注」，用户反馈「知道这周干什么」的比例明显提升

失败与修正：① 第一版把所有 AI 能力塞进 Chrome 扩展弹窗，自己用了两天就发现问题——弹窗空间太小，复杂分析结果放不下，关闭弹窗后什么也找不回来。推翻重做，把分析结果和管理能力迁到独立 Web 面板。② 手写各平台嗅探代码抓内容，B站/公众号/小红书纷纷失败，失败率极高。引入 defuddle 覆盖 30+ 平台做兜底——不是自己写一套完美的抓取，而是承认边界、用已有方案

成果：管线端到端验证通过，支持 B站、YouTube、公众号、GitHub、网页等主流信息源。为早期用户制作完整上手指南

问津 — 职业探索平台产品设计 + 独立全栈 2026.05 – 至今 · wenjin-4xc.pages.dev

匹配策略选型：为什么手动结构化而不是机器学习？评估过两种方案——用 LLM 实时匹配 vs 预结构化数据库 + 规则匹配。LLM 方案测试发现匹配结果不稳定，同一个用户重复测试给出不同职业推荐，且无法解释「为什么推荐这个」。选择手动构建 533 个职业的结构化数据库，每个附带匹配逻辑和可执行步骤，匹配结果可解释、可复现。代价是工作量，换来的是一致性和用户信任

提示词与算法调优：早期匹配只输出职业列表，用户反馈「看完还是不知道怎么办」。迭代后每个匹配结果附带 ① 你为什么适合的原因 ② 本周可执行的下一步 ③ 该职业的3年发展路径。从「告诉用户你是谁」变成「告诉用户你能做什么」

失败与修正：① 内容组织——第一版按功能模块平铺（测评/浏览/时间线），用户找不到入口。推翻后参照斯坦福人生设计课重构为六步叙事流（自画像 → 算法匹配 → 自由探索 → 5年时间线 → 行动清单 → 岔路口反思），每一步自然引导下一步。② 云同步方案——做过技术准备，但用户测试时发现职业焦虑场景下对隐私极度敏感，有人明确说「如果数据上传我就不填真实想法」。确认后彻底放弃云方案，所有数据只存 localStorage——不上传在此场景下反而是卖点

成果：六章体验全部上线，覆盖 533 个职业。用户可从「我是谁」走到「这周干什么」的完整闭环

Token 资源足迹 — 数据叙事网页独立完成 · 概念到上线 2026 · 7350final-4lt8.vercel.app

产品形态选型：为什么不做一个互动数据仪表盘？做过两个方案对比：A) 交互式仪表盘，用户自己探索数据；B) 单页滚动叙事，带用户走一条设计好的故事线。排除 A 的理由：目标用户是普通 AI 使用者，不是数据分析师——给他们仪表盘等于把认知负担丢回给用户。选 B 因为目标是「感知建立」，形式必须服从这个目标

换算策略与效果：把抽象 Token 消耗换算成日常单位（一瓶水、一度电、一个人工作几小时）。测试过纯数字展示 vs 类比换算——纯数字（如「1700 万升水」）用户看了无感，换算后（「够一个人喝 XX 年」）才有「wow」反应。关键设计判断：不是说教式地「AI 很费资源」，而是把事实放在那让用户自己得出结论

成果：从概念、数据整理、视觉设计到前端开发独立完成并上线。该项目的「形式选择要服从沟通目标」的方法论，可直接迁移至 AI 产品的用户引导和功能教育设计
